# Window Placement And Python UI Package Cleanup Design

日期：2026-03-26

## 背景

当前 Swift/AppKit shell 已经可以通过 `./icu` 启动，但桌宠窗口的初始位置仍然写死在 [`DesktopPetWindowController.swift`](/Users/clement/Workspace/icu/apps/macos-shell/Sources/ICUShell/Pet/DesktopPetWindowController.swift) 中的固定坐标。

这会带来两个问题：

1. 首次启动不满足“桌宠默认位于桌面右下角”的产品预期
2. 用户手动拖动后的窗口位置没有持久化，下次启动无法恢复

同时，用户还要求把之前旧 Python/Qt 启动链相关的本机全局依赖一起清理掉，但不能误伤 builder / 图像处理仍在使用的 Python 依赖。

## 目标

本次设计完成后，应满足以下条件：

1. 首次启动时，桌宠默认定位在当前可见屏幕区域右下角，并保留边距
2. 如果用户手动拖动过桌宠，则后续启动优先恢复上次位置
3. 窗口位置持久化进入现有 shell 运行态存储，不引入新的散落状态文件
4. 全局 Python 环境中的旧 Qt/UI 启动链相关包被移除
5. builder / 图像处理仍需要的 Python 包不被卸载

## 非目标

1. 本次不实现多显示器复杂策略，例如为每个显示器分别记忆位置
2. 本次不实现窗口吸附、磁贴、边缘动画等增强交互
3. 本次不卸载 builder / 图像处理仍需要的 Python 依赖
4. 本次不彻底删除仓库中的全部 legacy Python 文件

## 设计

### 1. 首次启动默认右下角

新增一个纯 runtime 的窗口定位规则：

1. 读取当前主屏幕或可用屏幕的 `visibleFrame`
2. 以窗口尺寸和固定边距计算右下角原点
3. 将该位置作为首次启动默认位置

边距只需使用一个简单常量，目标是让桌宠不贴边、不压住 Dock。

### 2. 拖动后记住位置

窗口位置存储进入现有 [`PersistedRuntimeState`](/Users/clement/Workspace/icu/apps/macos-shell/Sources/ICUShell/Runtime/ShellState.swift)：

1. 新增一个可选的窗口定位字段
2. 首次启动时该字段为空，因此使用右下角默认位置
3. 用户拖动窗口后，窗口控制器通过当前运行态存储更新该字段
4. 下次启动若该字段存在，则优先恢复该位置

这样可以复用已有的 `current_state.json`，避免再增加新的配置文件。

### 3. 恢复位置的安全边界

如果保存的位置明显超出当前可见区域，则不直接恢复，而是回退到右下角默认位置。

这样可以覆盖以下情况：

1. 分辨率变化
2. 显示器数量变化
3. 用户上次保存的位置已经不可见

### 4. 窗口控制器职责

窗口控制器需要做两件事：

1. 启动时根据“已保存位置 or 默认右下角位置”确定初始 frame
2. 在窗口移动后把新位置持久化

运行态计算尽量放在独立 helper 中，避免把几何计算散落在窗口控制器里。

### 5. 全局 Python UI 依赖清理

用户要求清理的是旧 Python/Qt 启动链相关的全局包，而不是所有 Python 包。

本次目标包为：

1. `PySide6`
2. `PySide6_Addons`
3. `PySide6_Essentials`
4. `shiboken6`
5. `rumps`（若存在）

这些包卸载前先检查当前 `pip3` 全局环境；如果本机里根本不存在，则如实记录“已不存在”，不伪造卸载结果。

### 6. 保留的 Python 依赖

以下类别不在本次卸载范围：

1. `requests`
2. `Pillow`
3. `opencv-python`
4. `rembg`
5. `huggingface-hub`
6. `transitions`
7. `watchdog`

原因是它们仍服务 builder、状态同步或迁移工具链。

## 备选方案比较

### 方案 A：每次启动都强制右下角

优点：

1. 逻辑最简单

缺点：

1. 用户手动拖动毫无意义
2. 体验会显得“窗口位置不受控”

### 方案 B：首次右下角，拖动后记住位置

优点：

1. 满足默认产品预期
2. 尊重用户调整后的布局
3. 持久化成本低

缺点：

1. 需要多一层窗口位置存储与恢复逻辑

本次采用方案 B。

### 方案 C：多显示器复杂记忆

优点：

1. 长期更完整

缺点：

1. 当前阶段超出必要复杂度
2. 容易拖慢迁移节奏

## 测试策略

### 运行态测试

验证以下内容：

1. 无保存位置时，会计算右下角默认坐标
2. 有保存位置且仍可见时，会恢复该位置
3. 保存位置不可见时，会回退到右下角默认坐标

### 启动 smoke test

验证以下内容：

1. `./icu` 启动后桌宠位置日志或行为符合预期
2. `~/Library/Application Support/ICU/state/current_state.json` 中出现窗口位置字段

### Python 包清理验证

验证以下内容：

1. 卸载前列出目标包的存在情况
2. 执行卸载命令
3. 卸载后再次确认目标包不再出现在 `pip3` 全局环境中

## 风险

1. 如果窗口位置直接写死在控制器内，后续多显示器或更多布局规则会变得难以维护。
2. 如果不做越界回退，用户在显示器变化后可能“丢失”桌宠窗口。
3. 如果误卸载 builder 依赖，会影响后续资产生成与迁移工作。
