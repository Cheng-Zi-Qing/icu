# Unified Theme System And Generation Routing Design

日期：2026-03-26

## 背景

当前 Swift/AppKit shell 已经成为默认运行路径，但 GUI 风格仍然是局部统一，而不是系统级统一：

1. 形象选择器和自定义形象向导已经有自己的像素风样式层
2. 桌宠右键菜单仍然使用系统 `NSMenu`
3. 状态栏菜单仍然直接绑定系统 `NSMenu`
4. 后续新增 GUI 目前没有强制接入统一主题 contract

这导致两个问题同时存在：

1. “像素风”还只是局部样式，不是产品默认主题
2. 后面如果要允许用户输入 vibe 并生成自己喜欢的主题风格，当前 GUI 架构没有统一承接点

同时，用户已经明确把未来的 AI 生成能力拆成三类业务域：

1. 主题
2. 形象与动画
3. 话术

但模型能力的配置方式不应该按这三个业务域硬拆，因为不同模型适配的场景本来就是按能力类型区分，而不是按产品业务模块区分。用户已确认未来模型配置页应按以下三类能力管理：

1. 文本描述
2. 动画形象
3. 代码生成

第一阶段需要先把 GUI 主题底座和能力型模型路由定下来，再进入后续 AI 生成体验。

## 目标

本次设计完成后，应满足以下条件：

1. 当前 Swift shell 的全部 GUI 都归属于同一个主题系统
2. 默认主题为 `像素风`
3. 桌宠右键弹出栏不再使用系统 `NSMenu`，而是使用可主题化的自绘弹层
4. 状态栏菜单不再直接依赖系统 `NSMenu`，而是使用可主题化的自绘弹层
5. 形象选择器、自定义形象向导、右键弹层、状态栏弹层、后续配置窗口都使用同一套主题 contract
6. 新增原生“生成配置页”，按 `文本描述 / 动画形象 / 代码生成` 三类能力配置模型
7. 第一阶段至少打通一条真实消费链，让主题系统开始消费能力型模型配置，而不是只保存静态配置
8. 后续用户输入 vibe 时，系统可以基于同一套能力配置继续扩展，而不需要返工 GUI 底座

## 非目标

1. 本次不直接实现完整的“输入 vibe 一键生成完整主题资产包”最终体验
2. 本次不实现多主题商店、主题市场或复杂主题管理器
3. 本次不重写整套话术结构
4. 本次不直接实现完整动画生成工作台
5. 本次不把所有遗留 Python 逻辑都并入新的生成路由
6. 本次不追求 1:1 复刻旧 Qt 所有视觉细节

## 用户确认的关键决策

1. GUI 必须被归类为统一主题系统，默认主题是像素风
2. 右键弹出栏也必须归入该主题系统，不能继续是系统风格例外
3. 未来要允许用户通过 vibe 生成自己喜欢的主题风格
4. 模型配置页必须按能力类型划分，而不是按主题/形象/话术业务域划分
5. 能力类型固定为：
   - 文本描述
   - 动画形象
   - 代码生成
6. 第一阶段不能只停留在“配置页保存”，而是要有至少一条真实消费链

## 方案比较

### 方案 A：只统一现有窗口主题，保留系统 `NSMenu`

优点：

1. 改动最小
2. 风险较低

缺点：

1. 无法满足“全部 GUI 统一主题”的目标
2. 右键弹出栏和状态栏菜单会永久成为系统风格例外
3. 后续主题生成时，菜单体系仍然需要返工

不采用。

### 方案 B：统一主题底座 + 自绘菜单弹层 + 能力型模型配置页 + 第一条真实路由

优点：

1. 能满足“默认像素风覆盖全部 GUI”的目标
2. 主题系统和未来 AI 生成链之间有真实接口，而不是空壳配置
3. 后续可以扩展到 vibe 生成主题，而不需要重写 GUI 结构

缺点：

1. 比单纯改样式多了一层模型配置和路由设计
2. 需要替换当前系统菜单实现

本次采用方案 B。

### 方案 C：主题底座、模型配置、vibe 生成器、主题包生成一轮做完

优点：

1. 概念完整

缺点：

1. 范围过大
2. 会把底座问题和生成器问题强行耦合
3. 第一阶段容易失控

不采用。

## 设计

### 1. 全局主题系统

当前只服务 avatar 面板的局部样式层应提升为全局主题系统。

建议引入两层结构：

1. `ThemeDefinition`
2. `ThemedComponents`

`ThemeDefinition` 是纯数据层，描述一个主题的 token。  
`ThemedComponents` 是渲染层，负责消费 token 并生成统一外观的 GUI 组件。

第一阶段默认只提供一个主题实现：

1. `PixelTheme`

但接口从第一天起就按“未来存在多个主题”来设计。

### 2. 主题 token 分层

主题 token 至少分成以下几类：

1. `ColorTokens`
   - 窗口背景
   - 卡片背景
   - 输入框背景
   - 菜单背景
   - 高亮色
   - 边框色
   - 主要文字色
   - 次级文字色
   - 危险色
   - 遮罩色

2. `TypographyTokens`
   - 标题字体
   - 正文字体
   - 小字字体
   - 菜单项字体
   - 状态标签字体

3. `SurfaceTokens`
   - 圆角
   - 边框宽度
   - 阴影
   - 内边距
   - 间距
   - 面板宽度策略

4. `MotionTokens`
   - 弹层出现动画
   - hover 高亮节奏
   - 选择切换动画
   - 面板淡入淡出

5. `AssetTokens`
   - 可选像素边框资源
   - 纹理
   - 装饰角标
   - 特殊背景图

这意味着未来“主题”不只是换颜色，而是能覆盖 GUI 的视觉语言与交互节奏。

### 3. 主题组件层

为了防止后续 GUI 再次出现各处散写样式，主题层必须配套一组可复用的基础组件。

第一阶段应至少收敛出这些组件：

1. `ThemedPanel`
2. `ThemedButton`
3. `ThemedTextField`
4. `ThemedTextView`
5. `ThemedSectionHeader`
6. `ThemedMenuPanel`
7. `ThemedMenuRow`
8. `ThemedSeparator`

现有的 avatar 选择器与向导应逐步改为消费这些基础组件，而不是继续维持一套只在 avatar 目录下可见的局部样式 helper。

### 4. 菜单与弹层统一

如果目标是“全部 GUI 统一主题”，则右键菜单和状态栏菜单必须从系统 `NSMenu` 迁出。

当前状态：

1. 桌宠右键菜单通过 `NSMenu.popUpContextMenu(...)`
2. 状态栏菜单通过 `statusItem.menu = menu`

这两种路径都无法实现真正统一的像素风主题。

因此第一阶段应改为：

1. `ContextMenuPanelController`
   - 服务桌宠右键弹出栏
   - 锚定到鼠标点击位置

2. `StatusMenuPanelController`
   - 服务状态栏按钮弹层
   - 锚定到状态栏按钮下方
   - 通过 `statusItem.button` 的点击事件显式拉起，而不是继续绑定 `statusItem.menu`

两者底层统一复用：

1. `ThemedMenuPanel`
2. `ThemedMenuSection`
3. `ThemedMenuRow`
4. `ThemedSeparator`

行为层不重写业务含义，而是继续复用已有菜单模型，例如当前的 `DesktopPetMenuModel`。  
也就是说，第一阶段改的是显示层和主题能力，不是重做业务动作。

为了避免自绘弹层行为分叉，第一阶段还应统一以下交互约束：

1. 点击菜单项后关闭弹层并执行动作
2. 点击弹层外部区域后关闭弹层
3. 按 `Esc` 可关闭弹层
4. 桌宠右键弹层与状态栏弹层共用关闭与事件监控策略

### 5. 第一阶段应覆盖的 GUI 范围

用户已明确要求“全部统一”，因此第一阶段应按“当前 Swift 已有 GUI 全量统一，未来 GUI 预留统一入口”来定义范围。

第一阶段实际落地覆盖：

1. 桌宠右键弹出栏
2. 状态栏弹层
3. 形象选择器
4. 自定义形象向导
5. 新增的生成配置页

第一阶段同步预留 contract，但不要求完整功能迁回：

1. 提醒气泡
2. 后续设置页
3. 后续新面板

要求是：

1. 新增 GUI 必须通过统一主题层接入
2. 不允许再出现新的系统风格例外组件

### 6. 能力型模型配置页

用户已明确否定“按主题/形象/话术分模型”的做法。  
因此模型配置必须按能力类型组织。

第一阶段新增一个原生 Swift/AppKit 配置窗口：

`GenerationConfigWindowController`

它只管理 3 类能力：

1. 文本描述
2. 动画形象
3. 代码生成

每类能力至少包含：

1. provider
2. endpoint / base URL
3. model
4. auth 信息
5. default options

存储位置统一落到 `config/settings.json`。

建议配置结构如下：

```json
{
  "generation": {
    "text_description": {
      "provider": "ollama",
      "base_url": "http://localhost:11434",
      "model": "qwen3.5:35b",
      "auth": {},
      "options": {
        "temperature": 0.7
      }
    },
    "animation_avatar": {
      "provider": "huggingface",
      "base_url": "https://api-inference.huggingface.co",
      "model": "stabilityai/stable-diffusion-xl-base-1.0",
      "auth": {
        "token": "..."
      },
      "options": {}
    },
    "code_generation": {
      "provider": "openai-compatible",
      "base_url": "https://...",
      "model": "...",
      "auth": {
        "api_key": "..."
      },
      "options": {}
    }
  }
}
```

这里的重点不是把所有 provider 一次写死，而是先把能力路由 contract 定住。

为了让第一条真实消费链有明确入口，`GenerationConfigWindowController` 第一阶段除能力配置外，还必须内置一个最小主题生成区：

1. 一个 `vibe` 文本输入区
2. 一个“生成并应用主题”动作
3. 一个“恢复默认像素风”动作
4. 一个最近一次生成结果/错误状态展示区

也就是说，第一阶段不额外新开一个主题实验窗口，而是直接把最小主题生成入口放进该配置页。

### 7. 主题生成的第一条真实路由

第一阶段不能只停留在“配置页可以保存”。  
必须打通至少一条真实消费链。

建议第一阶段先打通 `主题` 这一域的最小生成链，但路由方式仍然按能力类型组织：

1. `文本描述` 能力负责把用户的 vibe 扩展成结构化风格意图
2. `代码生成` 能力负责把这份风格意图归一化为 `ThemePack`
3. GUI 主题系统负责读取和应用 `ThemePack`

第一阶段不要求 `动画形象` 能力参与主题生成，但配置页必须已经具备这一栏位，以支撑后续形象与动画生成链。

也就是说，第一阶段的真实消费链是：

`vibe text -> StyleIntent -> ThemePack -> apply theme`

而不是：

`vibe text -> 直接拼接界面`

该链路第一阶段必须具备明确失败回退策略：

1. 任一能力配置缺失、鉴权失败、请求超时或返回非法结构时，不得影响应用启动
2. 生成失败时保留当前激活主题；若当前无有效主题，则回退到 `pixel_default`
3. 错误信息通过主题化 GUI 在配置页内展示，而不是只打印控制台

### 8. ThemePack contract

为了避免后续 AI 直接生成 Swift 代码，主题生成产物必须定义成结构化数据包：

```json
{
  "meta": {
    "id": "pixel_default",
    "name": "Pixel Default",
    "version": 1,
    "source_prompt": "..."
  },
  "tokens": {
    "colors": {},
    "typography": {},
    "surface": {},
    "motion": {},
    "assets": {}
  },
  "components": {
    "panel": {},
    "button": {},
    "menu_panel": {},
    "menu_row": {},
    "text_field": {}
  }
}
```

要求：

1. 主题生成器输出 `ThemePack`
2. 应用内有明确校验层验证 `ThemePack` 是否完整
3. GUI 渲染器只消费合法 `ThemePack`
4. 主题包是数据，不是代码

### 9. ThemePack 持久化与激活

第一阶段必须明确 `ThemePack` 的落盘与激活方式，否则 implementation plan 会被迫自行发明状态模型。

建议 contract 如下：

1. 生成后的 `ThemePack` 先经过校验，再写入 `AppPaths.stateDirectory/themes/<theme-id>.json`
2. 当前激活主题 ID 写入 `config/settings.json`
3. 应用启动时先读取激活主题 ID，再从主题存储目录加载对应 `ThemePack`
4. 若主题文件不存在、解析失败或校验失败，则回退到 `pixel_default`
5. `pixel_default` 作为内建主题始终可用，不依赖外部文件存在

这样可以把“模型配置”与“运行时主题产物”分开：

1. `config/settings.json` 保存用户配置与当前激活主题引用
2. `AppPaths.stateDirectory/themes/` 保存生成出来的主题数据包

### 10. 运行时主题应用

第一阶段虽然默认只提供 `PixelTheme`，但架构上应允许后续切换主题。

最小运行机制：

1. 启动时读取当前主题 ID
2. 若不存在则回退到 `pixel_default`
3. 弹层、面板、按钮、输入框统一从当前主题解析 token
4. 未来当主题切换发生时，已打开 GUI 至少应支持重建或重新渲染

第一阶段不要求实现完整的多主题切换体验，但必须确保默认主题不再是“散落在各窗口里的局部常量”。

### 11. 与其他 AI 生成域的关系

本次主题系统设计不是只服务主题本身，而是为后续三类生成域提供统一的能力路由基础。

业务域仍然保持：

1. 主题
2. 形象与动画
3. 话术

但模型配置与执行路由按能力类型组织：

1. 文本描述
2. 动画形象
3. 代码生成

这样后续扩展时：

1. 主题会消耗 `文本描述 + 代码生成`
2. 形象与动画会消耗 `文本描述 + 动画形象`
3. 话术会主要消耗 `文本描述`

用户看到的是业务功能；系统底层走的是能力路由。

## 第一阶段落地边界

第一阶段必须完成：

1. 全局主题 contract
2. 默认 `PixelTheme`
3. 右键弹层替换
4. 状态栏弹层替换
5. 现有 Swift GUI 全量接入主题系统
6. 原生生成配置页
7. `config/settings.json` 中的能力型模型配置
8. 配置页内的最小主题生成入口
9. `ThemePack` 的持久化与激活机制
10. 主题的第一条真实消费链

第一阶段明确不做：

1. 多主题库管理器
2. 主题市场
3. 完整动画生成工作台
4. 话术 schema 大改
5. 一次性统一全部遗留 Python UI

## 验证策略

实施完成后，应至少满足以下验证条件：

1. 所有 Swift GUI 不再直接散写局部样式，而是统一经主题层取值
2. 桌宠右键弹出栏不再使用系统 `NSMenu`
3. 状态栏菜单不再直接依赖系统 `NSMenu`
4. 默认主题为 `PixelTheme`
5. 生成配置页可独立保存三类能力配置
6. 配置页内可直接输入 `vibe` 并触发“生成并应用主题”
7. 主题生成链已真实消费 `文本描述` 与 `代码生成` 两类配置
8. 生成出的合法 `ThemePack` 可持久化并在重启后重新加载
9. 生成失败或配置缺失时，应用保持可用并回退到当前主题或 `PixelTheme`
10. `./icu --verify` 继续通过
11. 新增主题/弹层相关手工 runtime 校验

## 风险与边界提醒

1. 自绘弹层替换系统菜单后，关闭逻辑、键盘交互和窗口层级管理会变复杂
2. 如果第一阶段不提前冻结 `ThemePack` contract，后续 AI 生成主题时很容易返工
3. 如果主题组件层不及时抽象，现有窗口可能继续复制局部样式，导致“统一主题”在实现上再次破功
4. 配置页如果按业务域而不是能力域设计，会直接违背用户已确认的模型配置逻辑

## 结论

第一阶段采用以下方向：

1. 建立全局主题系统
2. 默认主题固定为像素风
3. 用自绘弹层替换右键与状态栏菜单
4. 新增按能力类型组织的生成配置页
5. 用 `文本描述 + 代码生成` 打通主题系统的第一条真实消费链

这样可以在不把第一阶段做爆的前提下，同时满足：

1. GUI 默认像素风统一
2. 右键弹出栏纳入同一主题体系
3. 后续 vibe 生成主题可持续扩展
4. 模型配置逻辑与用户确认的能力分组保持一致
