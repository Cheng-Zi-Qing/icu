# Inline Avatar Creation Flow Design

**Date:** 2026-03-30

**Status:** Approved in browser and terminal review

## Goal

把当前 `新增自定义形象` 这条仍然依赖旧三步向导的主流程，收口到已经建立好的 `更换形象` 三 tab 工作台中：

- 不再弹出独立旧向导窗口
- 不再引入第四个 tab
- 保持 `主题风格 / 桌宠形象动画 / 话术` 的职责边界不被打乱
- 让自定义形象生成也遵循 `生成 -> 预览 -> 不满意再生成 -> 满意再应用`

这份设计的目标不是重新设计一套新建系统，而是让“新建自定义形象”成为现有工作台里的一个内联创作态。

## Current Context

仓库当前已有以下基础：

- `AvatarSelectorWindowController` 已经承载 `主题风格 / 桌宠形象动画 / 话术` 三个 tab
- `新增自定义形象` 目前仍通过 `AvatarCoordinator.presentAvatarWizard()` 打开旧 `AvatarWizardWindowController`
- 自定义形象底层落盘需求已经非常简单：
  - 名称
  - persona 文本
  - `idle / working / alert` 三张动作图
- 落盘仍然复用现有 `AvatarAssetStore.saveCustomAvatar(...)`

这意味着要收口的是主流程和 UI 结构，而不是推翻底层资产模型。

## Product Decisions

### 1. 新建形象不再是独立窗口

`新增自定义形象` 不再触发旧向导窗口。

用户从 `更换形象 > 桌宠形象动画` 点击 `新增自定义形象` 后，仍停留在当前窗口中，只把当前 tab 从“浏览态”切换为“新建创作态”。

这条决策的目的很明确：

- 保持同一工作台心智模型
- 避免出现“双轨 UI”
- 让后续模型配置、预览、应用逻辑都能和现有 tab 体系一致

### 2. 三 tab 结构保持不变

本次设计不新增第四个 tab。

`更换形象` 继续保持三个互不混职的 tab：

- `主题风格`
- `桌宠形象动画`
- `话术`

其中：

- `主题风格` 只负责 GUI / 菜单 / 气泡等视觉主题
- `桌宠形象动画` 只负责桌宠外观与动作资产
- `话术` 只负责文案人格

“新建自定义形象”属于 `桌宠形象动画` tab 内部的一种工作状态，而不是新的产品域。

## Avatar Tab Mode Model

### 1. 桌宠形象动画 tab 拆成两种模式

`桌宠形象动画` tab 固定为两种模式：

- `browse`
- `create`

默认进入 `browse`。

### 2. Browse 模式保持现有结构

`browse` 模式沿用当前已存在的信息架构：

- 左侧：形象列表
- 右侧：预览与说明
- 底部：预览 / 重新生成 / 应用 / 新增自定义形象

这部分不因本次改造而被整体推翻。

### 3. Create 模式是内联创作态

点击 `新增自定义形象` 后，当前 tab 进入 `create` 模式：

- 左侧仍保留现有形象列表，作为参考与返回锚点
- 右侧不再显示“现有形象详情”，而是切换为创作工作区

这样做的原因：

- 用户仍知道自己处在“形象”这一层
- 不会丢失已有形象参考
- 不需要再理解“进入另一个系统”

## UI Layout

### 1. 左栏

左栏继续保留现有形象列表，但顶部新增轻量模式头部：

- `当前模式：新建形象`
- 一个次级按钮：`返回现有形象`

左栏的职责只有两个：

- 作为参考列表
- 作为退出创作态的视觉锚点

不在左栏堆叠额外字段。

### 2. 右栏只保留三块

`create` 模式下，右栏固定为三段式结构，从上到下严格分层：

1. `Prompt 区`
2. `动作预览区`
3. `保存信息区`

这三块分别对应三种不同任务：

- 输入
- 看结果
- 决定是否保存

避免再次出现“所有信息堆在一个详情页里”的问题。

### 3. Prompt 区

Prompt 区只包含两项：

- `原始 prompt`
- `优化后 prompt`

语义与主题页一致：

- 原始 prompt 是用户输入源
- 优化后 prompt 是后续生成动作预览的真实输入源

### 4. 动作预览区

动作预览区固定为三个卡槽：

- `idle`
- `working`
- `alert`

每个卡槽只承担一个责任：显示对应动作预览。

不在这里再堆叠大量说明文案。说明性交互由状态条承担。

### 5. 保存信息区

保存信息区只包含：

- `形象名称`
- `persona`

名称是保存前的必填信息。

persona 允许系统先给出初稿，用户可以继续手动编辑。

## Interaction Flow

### 1. 主流程固定

`create` 模式的主流程固定为：

1. 输入原始 prompt
2. 点击 `优化 prompt`
3. 得到优化后 prompt
4. 点击 `生成预览`
5. 生成 `idle / working / alert` 三态预览
6. 如不满意，点击 `重新生成`
7. 满意后补全或修改 `名称 / persona`
8. 点击 `保存并应用`

### 2. 不允许自动保存

本设计明确禁止“生成后自动保存并切换”。

自定义形象必须和主题风格一样遵循：

- 先生成
- 再预览
- 满意后才应用

这是当前产品已经被用户明确确认的核心交互原则。

### 3. 取消行为

点击 `取消` 或 `返回现有形象` 时：

- 退出 `create` 模式
- 回到 `browse` 模式
- 不写入任何资产
- 不切换当前桌宠

## State Model

### 1. Create 模式最小状态

`create` 模式至少需要以下状态：

- `rawPrompt`
- `optimizedPrompt`
- `generatedActionImageURLs`
- `draftName`
- `draftPersona`
- `creationStage`

其中 `creationStage` 最小取值集合为：

- `empty`
- `drafted`
- `previewReady`
- `saving`

### 2. Preview Ready 的成立条件

只有当以下条件都成立时，`creationStage` 才能进入 `previewReady`：

- 已存在非空优化后 prompt
- 已成功生成 `idle`
- 已成功生成 `working`
- 已成功生成 `alert`

### 3. Save And Apply 的前置条件

`保存并应用` 只有在以下条件满足时才可用：

- 三态预览齐全
- 名称非空
- 当前不处于 `saving`

persona 允许为空，但推荐系统自动提供初稿再让用户修改。

## Error Handling

### 1. 优化失败

当 `优化 prompt` 失败时：

- 不清空原始 prompt
- 不覆盖已有优化稿
- 不退出创作态
- 状态条显示错误
- 允许立即重试

### 2. 生成预览失败

当 `生成预览` 或 `重新生成` 失败时：

- 如果此前没有成功预览，则保持空白预览态
- 如果此前已经有成功预览，则保留最近一次成功预览
- 不写入资产目录
- 不自动退出创作态

### 3. 保存失败

当 `保存并应用` 失败时：

- 保留当前 prompt、三态预览、名称和 persona 草稿
- 停留在当前创作态
- 允许用户修正后再次保存

## Integration Boundaries

### 1. 入口迁移

`AvatarCoordinator` 的主入口切换为：

- `新增自定义形象` 不再调用 `presentAvatarWizard()`
- 改为驱动 `AvatarSelectorWindowController` 进入 `create` 模式

### 2. 底层落盘复用现有实现

保存成功时仍然调用现有：

- `AvatarAssetStore.saveCustomAvatar(...)`
- `AvatarSettingsStore.saveCurrentAvatarID(...)`

也就是说，本次不引入新的资产格式。

### 3. 旧向导退场策略

迁移顺序固定为：

1. 主入口切换到内联创作态
2. 创作态验证通过
3. 旧 `AvatarWizardWindowController` 从主流程中移除

旧向导不再是用户主流程的一部分。

## Testing Strategy

本次实现前应先补行为测试，再改生产代码。

至少覆盖以下场景：

1. 点击 `新增自定义形象` 后，不再打开旧向导，而是在当前 tab 渲染 `create` 模式
2. `create` 模式必须走 `优化 prompt -> 生成预览 -> 保存并应用`
3. 缺少任一动作预览时，`保存并应用` 不可用
4. 点击 `取消` 后回到 `browse` 模式，且不落盘
5. 保存成功后：
   - 新形象进入形象列表
   - 当前桌宠立即切换到新形象

## Non-Goals

本次设计不处理以下内容：

- 把主题和话术创作合并进形象创作态
- 新增第四个 tab
- 重新设计底层资产格式
- 立即删除全部旧向导代码而不经过主流程切换验证
